## What are Model-Driven Apps?

### The Concept

Model-driven apps are built on top of your **data model** in Dataverse. Unlike canvas apps where you design every pixel, model-driven apps automatically generate the UI based on your tables, columns, relationships, and business logic.

### Key Characteristics

| Feature | Description |
|---------|-------------|
| **Data-first design** | Build the data model, UI follows |
| **Responsive** | Automatically adapts to devices |
| **Role-based** | Security roles control visibility |
| **Consistent UX** | Same navigation, look, feel |
| **Low maintenance** | Changes to model reflect in UI |

### When to Use Model-Driven Apps

| Scenario | Best Choice |
|----------|-------------|
| Complex data relationships | Model-Driven âœ“ |

| Multiple forms for same data | Model-Driven âœ“ |
| Highly custom UI needed | Canvas |
| External data (non-Dataverse) | Canvas |
| Simple single-purpose app | Canvas |

### Multiple Forms

You can have **multiple Main forms** for different scenarios:

| Form Name | Purpose | Assigned To |
|-----------|---------|-------------|
| Sales Contact Form | Fields for sales reps | Sales security role |
| Support Contact Form | Fields for support | Support security role |
| Admin Contact Form | All fields | System Administrator |

**Form Order and Selection:**
- Users see the first form they have access to
- Fallback forms provide access if no specific form assigned
- Form order can be configured in the form designer

### View Types

#### System Views

Created by administrators, shared with entire organization.

| View Type | Description | Example |
|-----------|-------------|---------|
| **Public Views** | Available to all users | "Active Accounts" |
| **Quick Find View** | Used for search | Searches Name, Email, Phone |
| **Lookup View** | Shown in lookup dialogs | Minimal columns |
| **Associated View** | Related records subgrid | Contacts for this Account |
| **Advanced Find View** | Default for Advanced Find | All columns available |

#### Personal Views

Created by individual users for their own use.

- Only visible to creator
- Cannot be shared (use Public View instead)
- Created via "Save as Personal View"

## Business Rules

### What are Business Rules?

Business rules provide **no-code logic** for forms without writing JavaScript.

**Capabilities:**
- Show/hide fields
- Enable/disable fields
- Set field values
- Set field requirement levels
- Show error messages
- Set default values
- Lock/unlock fields

### Business Rule Scope

**Critical Exam Topic!**

| Scope | Executes On | Use Case |
|-------|-------------|----------|
| **Table (Entity)** | Server-side (Dataverse) | Canvas apps, all forms |
| **All Forms** | Client-side (browser) | All model-driven forms |
| **Specific Form** | Client-side (browser) | Only that form |

### Business Rule Examples

**Example 1: Show/Hide Based on Selection**
```
IF Account Type = "Enterprise"
THEN
  - Show "Credit Limit" field
  - Set "Credit Limit" as Required
ELSE
  - Hide "Credit Limit" field
```

**Example 2: Auto-Calculate Value**
```
IF Discount Percentage changes
THEN
  - Set "Discounted Price" = Price * (1 - Discount Percentage)
```

**Example 3: Validation Message**
```
IF End Date < Start Date
THEN
  - Show Error on End Date: "End date must be after start date"
```

| Requirement | Use |

| Real-time calculations | Formula Columns or Business Rules |

## Common Exam Scenarios

### Scenario 1: Form Selection
**Question:** "Different security roles need different forms. Sales sees minimal fields, Admin sees all fields."

**Answer:** Create multiple Main forms, assign each to different security roles, configure form order.

### Scenario 2: Displaying Parent Data
**Question:** "On the Opportunity form, show Account's industry and revenue without navigation."

**Answer:** Create a Quick View form on Account, add Quick View control to Opportunity form.

### Scenario 3: Rapid Data Entry
**Question:** "Users need to quickly add contacts without leaving the Account view."

**Answer:** Create a Quick Create form for Contact, enable Quick Create on Contact table.

### Scenario 4: Business Rule Scope
**Question:** "A business rule needs to work with both model-driven and canvas apps."

**Answer:** Set scope to **Table (Entity)**. Note: Only server-side actions (Set Value, Set Required) will work.

### Scenario 5: Interactive Filtering
**Question:** "Dashboard where clicking a chart segment filters all lists."

**Answer:** Interactive Dashboard with visual filters.

### Scenario 6: View Creation Without Code
**Question:** "Create a view showing contacts created in the last 30 days, no custom code allowed."

**Answer:** Use the **Maker Portal** to create a system view with a date filter.

---

## Key Takeaways for the Exam

1. **Four form types**: Main, Quick Create, Quick View, Card - know when to use each
2. **Business rule scope matters**: Entity = server-side, Form = client-side
3. **Canvas apps only use Entity-scoped** business rules
4. **Quick Create must be enabled** at the table level
5. **Quick View shows related record data** read-only
6. **Card forms are for dashboards** and compact display
7. **Views are saved queries** - columns, filters, sorting
8. **Site map = navigation structure** (Areas > Groups > SubAreas)
